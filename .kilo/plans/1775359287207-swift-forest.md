# Dynamic Provider Configuration in UI

## Goal
Allow users to add/configure model providers through the onboarding and settings pages, without editing code. Support provider-specific options like `enable_thinking` for reasoning models.

## Architecture

### Provider Types (Templates)
Each provider type defines how to call the API and what options it supports:

| Type | Client | Special Options |
|------|--------|-----------------|
| `openai_compatible` | OpenAI client with custom base_url | `extra_body`, `enable_thinking` |
| `anthropic` | Native Anthropic SDK | None |
| `gemini` | Google genai SDK | None |
| `zhipu` | Zhipu SDK | None |

### Storage
- `providers.json` - Custom providers added by user
- Format:
```json
{
  "providers": {
    "nvidia": {
      "name": "NVIDIA NIM",
      "type": "openai_compatible",
      "base_url": "https://integrate.api.nvidia.com/v1",
      "env_key": "NVIDIA_API_KEY",
      "models": ["z-ai/glm4.7"],
      "options": {
        "enable_thinking": true,
        "extra_body": {"chat_template_kwargs": {"enable_thinking": true}}
      }
    }
  }
}
```

## Implementation Plan

### Phase 1: Backend - Provider Registry

1. **Create `provider_registry.py`**
   - Load built-in providers from `model_router._PROVIDERS`
   - Load custom providers from `providers.json`
   - CRUD functions: `list_providers()`, `add_provider()`, `update_provider()`, `delete_provider()`
   - Merge both sources for `get_all_providers()`

2. **Add API endpoints in `telegram_bot.py`**
   - `GET /api/providers` - List all providers (built-in + custom)
   - `POST /api/providers` - Add new provider
   - `PUT /api/providers/{name}` - Update provider
   - `DELETE /api/providers/{name}` - Delete provider
   - `POST /api/providers/{name}/fetch-models` - Fetch models from provider API

3. **Modify `model_router.py`**
   - Import `provider_registry` 
   - `resolve_model()` checks both built-in and custom providers
   - Add `call_openai_compatible()` that uses base_url from provider config
   - Pass `extra_body` options when calling the API
   - Handle `reasoning_content` in streaming responses

### Phase 2: UI - Onboarding Page

1. **Add "Custom Provider" option to provider grid**
   - New card: "Add Custom Provider" with + icon
   - Opens modal/form for configuration

2. **Provider configuration form**
   - Name (display name)
   - Type dropdown: OpenAI Compatible, Anthropic, Gemini, Zhipu
   - Base URL (for OpenAI Compatible)
   - API Key input
   - Model selection (auto-fetch or manual entry)
   - Options section (collapsible):
     - Enable Thinking checkbox
     - Extra Body JSON editor (advanced)

3. **Save provider to backend**
   - Call `POST /api/providers`
   - Store API key in `.env` file
   - Refresh model dropdowns

### Phase 3: UI - Settings Page

1. **Add "Providers" section**
   - List all configured providers
   - Edit/Delete buttons for custom providers
   - "Add Provider" button

2. **Provider cards showing:**
   - Name, type, base URL
   - Models count
   - Options enabled (e.g., "Reasoning: enabled")
   - API key status (set/unset)

3. **Edit modal same as onboarding**

### Phase 4: Reasoning Model Support

1. **Backend changes to `model_router.py`**
   - Detect if provider has `enable_thinking` option
   - Add `extra_body` to API call when enabled
   - Parse `reasoning_content` from streaming response
   - Return reasoning in response dict: `{"content": ..., "reasoning": ..., "tool_calls": ...}`

2. **UI changes in `core.py`**
   - If response has `reasoning`, format it differently (gray text, collapsible)
   - Stream reasoning tokens in real-time with different color
